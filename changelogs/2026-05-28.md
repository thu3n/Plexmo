# Changelog - 2026-05-28

- **Drop unused `cron-parser` and `cronstrue` npm deps.** Followup to the library-sync removal. With no scheduled jobs left in cron.ts and no JobRow / cron-formatting utils in the settings page, both packages were zero-usage. `npm uninstall cron-parser cronstrue` removed 44 transitive packages (cron-parser pulled luxon and a chunky tree). Production build verified working after `rm -rf .next && npm run build` — Next.js had stale type-validator artifacts from the deleted routes that needed wiping. _Affected: `package.json`, `package-lock.json`._ Category: `chore`
- **Remove the library-sync feature entirely.** Followup investigation showed nothing in the UI actually rendered the `libraries` data — `getDashboardSnapshot` fetched it, the dashboard hook destructured it, and no component ever read it. The `/libraries/[key]` browse page existed but had no inbound links. So the whole stack came out: `syncAllLibraryLists` (the nightly 4 AM cron entry the user noticed), `syncLibraries`, `getLibraries`, `syncLibraryItems`, `getLibraryItems`, `getLibraryItemsPaginated`, `fetchLibraries` — entire `src/lib/libraries.ts` deleted (320 lines). Deleted orphan routes: `/api/libraries/[key]/items`, `/api/setup/test_sync`, `/api/history/export/test-sync`, and the `/libraries/[key]` page itself. `/api/jobs` POST handler emptied (no callers left); only GET (job history) remains since Tautulli import still creates jobs for its progress bar. Cron block for scheduled library-list sync removed; with no scheduled jobs left, the cron-parser dynamic-import dance disappeared too (~50 lines of bundler-workaround code gone). Settings → Jobs page restructured: scheduled-jobs table + JobRow + CronControl + utils.ts (cron formatting) all deleted; page now hosts only the History Repair panel, Link History button, and Tautulli import execution history. **Database**: `libraries` and `library_items` tables dropped from migrations.ts v1 baseline. `user_stats.calculateStreaks` had its `LEFT JOIN library_items` removed (the `libraryMeta` fallback for stream-duration was minor and already running on stale data). `servers.ts` deleteServerData transaction no longer cascades to `libraries`/`library_items`. `db-types.ts` `LibraryRow` + `LibraryItemRow` interfaces dropped. `LibrarySection` type removed from `plex-types.ts`. `getRunningJobForTarget` removed from jobs.ts (no callers). _Live DB cleanup_: drop `libraries` + `library_items` tables, drop `job_sync_libraries_*` orphan settings, VACUUM. Affected: 16 files changed, ~600 lines removed net. **What works on a new install**: clean Settings → Jobs page with Maintenance Tools + Execution History; Tautulli import keeps its progress bar; dashboard unaffected. Cron-parser and cronstrue npm deps are now zero-usage (can drop in a follow-up). _Affected: `src/lib/libraries.ts` (deleted), `src/app/libraries/[key]/page.tsx` (deleted), `src/app/api/libraries/[key]/items/route.ts` (deleted), `src/app/api/setup/test_sync/route.ts` (deleted), `src/app/api/history/export/test-sync/route.ts` (deleted), `src/app/settings/jobs/CronControl.tsx` + `utils.ts` (deleted), `src/lib/cron.ts`, `src/lib/jobs.ts`, `src/lib/migrations.ts`, `src/lib/user_stats.ts`, `src/lib/servers.ts`, `src/lib/dashboard-cache.ts`, `src/lib/db-types.ts`, `src/lib/plex/plex-library.ts`, `src/lib/plex/plex-types.ts`, `src/lib/plex/index.ts`, `src/app/api/jobs/route.ts`, `src/app/api/dashboard/route.ts`, `src/app/settings/jobs/page.tsx`, `src/features/dashboard/hooks/useDashboardData.ts`._ Category: `refactor`
- **Remove `sync_all_content` job + orphan DB cleanup.** Investigation of the live DB found one `sync_all_content` row with status `running` stuck at `updatedAt 2026-01-15` — the duplicate-prevention check (`getRunningJobForTarget`) had been blocking every subsequent 3 AM cron tick for 4+ months. Nobody noticed because the features that read `library_items` (HistoryRepairPanel, rewrap metadata, parts of `user_stats`) degrade gracefully on stale data. Rather than fix the stuck job, removed the entire `sync_all_content` job type: deleted `syncAllLibrariesContent` from `libraries.ts`, the first scheduled-job block in `cron.ts` (job_sync_content_*), the `sync_all_content` branches in `/api/jobs` POST and `/api/history/export/test-sync`, and the "Global Content Sync" `JobRow` from the Jobs settings page. The remaining "Sync Library Lists" row now has its previously-missing `onStart` and `onEnabledChange` handlers wired up (the Run button was a no-op before). The `jobs` table, `library_items` table, and the entire `sync_all_library_lists` flow stay intact — Tautulli import still uses jobs for its progress bar, and `sync_all_library_lists` continues running nightly at 4 AM. Live DB cleanup ran in one transaction: dropped the stuck job row, dropped the three orphan `job_sync_content_*` settings keys, and (broader audit finding) dropped 7,415 `library_items` rows belonging to a server that no longer exists in the `servers` table — `WHERE serverId NOT IN (SELECT id FROM servers)`. Followed by `VACUUM` to reclaim disk. Stale-but-not-orphan `library_items` rows for current servers (~28,684 rows) deliberately retained — they still serve as a local cache even if nothing refreshes them. _Affected: `src/lib/libraries.ts`, `src/lib/cron.ts`, `src/app/api/jobs/route.ts`, `src/app/api/history/export/test-sync/route.ts`, `src/app/settings/jobs/page.tsx`._ Category: `refactor`
- **Squash migrations v1-v4 into a single clean baseline.** With confirmation that nobody beyond the maintainer has pulled the post-main `:latest` image, collapsed the migration history into one entry. The new `migrations.ts` is 273 lines (was 454), has a single `v1: baseline_schema` migration that creates the final-state schema directly — plain `CREATE TABLE` (no `IF NOT EXISTS`), no `addColumnIfMissing` helper, no defensive idempotency, no `library_groups*` drop (those tables simply never get created). Removed the `addColumnIfMissing` helper entirely. New contributors reading the migration file see a single clean schema definition instead of scar-tissue accumulation from the main-era stamp + dropped-feature drop. _Migrator note: the maintainer's running database had its `schema_migrations` ledger manually reset to `(1, baseline_schema)` before this deploy so the new code's version-gate sees the schema as already-applied (the live DB shape already matches the new v1 final state from the previous deploy)._ _Affected: `src/lib/migrations.ts`._ Category: `refactor`
- **Fix migration v1 sequencing bug + Logger swallowing Error.message.** During live deploy onto a pre-existing `main`-era DB, v1 failed with `SQLITE_ERROR` because three `CREATE INDEX` statements (`idx_history_plex_guid`, `idx_history_imdb`, `idx_history_tmdb`) sat inside the initial `db.exec` block, *before* the `addColumnIfMissing` ALTERs that add the columns they reference. On a fresh DB this works (columns are present from `CREATE TABLE`); on a pre-existing DB missing those columns it doesn't. Moved the three index creations into a separate `db.exec` block *after* the column-additions. Also fixed `Logger.formatMessage` — it was passing `Error` objects through `JSON.stringify`, which only serializes enumerable own properties; `.message` and `.stack` are not enumerable, so the full error was being silently dropped from production logs (we only saw `{"code":"SQLITE_ERROR"}` with no message, which is what made this bug hard to diagnose). Logger now special-cases `Error` to print message + stack. _Affected: `src/lib/migrations.ts`, `src/lib/logger.ts`._ Category: `fix`
- **Migration v4 drops legacy `library_groups` / `library_group_members` tables.** The unified-libraries feature was removed from `refactor` (route handlers, pages, lib helper, settings UI all deleted), but the two SQLite tables that backed it are still present on databases that were created under `main`'s embedded-CREATE-TABLE init code. Added `version: 4, name: "drop_library_groups"` to `migrations.ts` — drops the child table first (it has a FOREIGN KEY on the parent), `IF EXISTS` so refactor-era fresh installs are a clean no-op. Runs automatically on next container start. Codified rather than hand-run via `sqlite3 .drop` so any future `main`-era database picked up by this codebase is auto-cleaned the same way. _Affected: `src/lib/migrations.ts`._ Category: `refactor`
- **Unreachable-server indicator + auto-cache-invalidation on server save/delete.** Building on the in-memory dashboard cache: `runCronJob` now calls `markServerFailure(server.id, message)` from `dashboard-cache` whenever a per-server fetch throws, so the cache distinguishes "no data yet" from "tried recently and failed." The aggregated `/api/dashboard` branch was rewritten to be strictly read-only — it never prime-fetches on a request thread anymore, which was the actual cause of the user-reported 10s stalls (one unreachable server in `Promise.allSettled` was blocking every dashboard refresh for the full 10s plexFetch timeout). The single-server branch keeps prime-on-miss but honours a 30s failure TTL so it doesn't repeatedly stall on dead servers either. **`/api/servers` enriches each entry with `status: "ok" | "unreachable"`** (and `statusMessage`) derived from `getServerFailure` + `getServerSnapshot` — a server is reported unreachable only when cron failed AND no cached snapshot exists. **Filter chips on the dashboard** (both desktop pill row and mobile horizontal scroll in `src/app/page.tsx`) now render a small amber warning triangle next to the server name when `status === "unreachable"`, with the failure message exposed via the native `title` tooltip. Reuses the same triangle SVG as the rule-violation badge in `SessionCard.tsx` (different colour to distinguish). **Server save/delete now invalidates and re-primes immediately**: POST `/api/servers`, PUT `/api/servers/[id]` and DELETE `/api/servers/[id]` all call `deleteServerSnapshot(id)`; POST/PUT additionally fire-and-forget `runCronJob()` so the dashboard reflects new credentials/URL within ~1s instead of waiting up to 60s for the next scheduled tick. `PublicServer` type gained optional `status` and `statusMessage` fields. _Affected: `src/lib/dashboard-cache.ts`, `src/lib/cron.ts`, `src/lib/servers.ts`, `src/app/api/dashboard/route.ts`, `src/app/api/servers/route.ts`, `src/app/api/servers/[id]/route.ts`, `src/app/page.tsx`._ Category: `feature`
- **In-memory dashboard snapshot cache; `/api/dashboard` no longer fetches Plex live.** Previously every dashboard poll (SWR, every 5s, every open client) ran `Promise.allSettled(servers.map(getDashboardSnapshot))` — `1 + N` HTTP requests to Plex per server (one `/status/sessions` XML + one metadata fetch per active stream), with a 10s per-request timeout. Each connected browser re-paid the cost. The fix wires a shared per-server cache the existing background worker keeps warm: the 60s cron and the WebSocket listener (which already triggers cron on every Plex play-state change with a 2s cooldown) now also call `setServerSnapshot(server.id, snapshot)` after each fetch. `/api/dashboard` reads from that cache via a `Map` lookup; cold-start requests use a prime-on-miss fallback so the first hit after server boot is still correct rather than empty. Aggregated `updatedAt` reflects the oldest per-server `cachedAt` so the client can see freshness. No DB schema change — session state is ephemeral and the authoritative source is Plex; persisting it would only let stale snapshots survive restart. Cache survives Next.js hot reload via `globalThis.__plexmo_dashboard_cache`, matching the existing pattern in `instrumentation.ts`. Verified: lint clean on new/touched files (pre-existing `any` warnings in `cron.ts` untouched), all 7 vitest tests pass. _Affected: `src/lib/dashboard-cache.ts` (new), `src/lib/cron.ts`, `src/app/api/dashboard/route.ts`._ Category: `refactor`
- **Tuned SQLite for `activity_history` aggregate hot path instead of switching engines.** After a database-choice review concluded that the single-container deployment story doesn't justify a Postgres or DuckDB migration (and `better-sqlite3`'s synchronous API is deeply embedded), addressed the actual concern — dashboard aggregate query speed as history grows — entirely inside SQLite. **(1) Migration v2 `history_user_stoptime_indexes`** adds `idx_history_userid_stoptime` and `idx_history_user_stoptime` so `getStatsForPeriod` / `getRecentlyPlayed` / `getPlayerStats` stop full-scanning. The pre-existing `idx_history_dup_check (user, ratingKey, ...)` couldn't satisfy `WHERE user=? AND stopTime>?` because of column order. Two indexes because the `WHERE user OR userId` shape uses SQLite's MULTI-INDEX OR (one index per branch — verified via `EXPLAIN QUERY PLAN`). **(2) Migration v3 `summary_table_and_retention`** promotes `player` from `json_extract(meta_json,'$.player')` to a real column (with one-shot backfill) so player-stats GROUP BY can use the new indexes; creates `user_activity_summary` (per-user totals, keyed on `COALESCE(userId, user)`) backfilled from existing history; adds `idx_concurrent_snapshots_timestamp`, `idx_rule_events_triggered_at`, `idx_jobs_status_updated` for the retention sweep. **(3) `getAllTimeStats` now reads from `user_activity_summary`** — O(1) primary-key lookup instead of a full history scan. All four user-stats hot queries verified via `EXPLAIN QUERY PLAN` to use the new indexes (`SEARCH ... USING INDEX ...` instead of `SCAN activity_history`). **(4) History insert path consolidated** through a new `insertHistoryRow()` helper that updates `user_activity_summary` atomically with each insert (used by `syncHistory`, `addHistoryEntry`, and the Tautulli import). `last_played_at` uses `MAX(...)` to stay monotonic against out-of-order Tautulli imports. `deleteHistory` rebuilds affected user-buckets from `activity_history` after delete; `deleteAllHistory` truncates the summary in the same transaction. **(5) New `src/lib/retention.ts`** runs a daily-gated sweep from `cron.ts`: prunes `concurrent_snapshots` (default 90 days), `rule_events` (180 days), finished `jobs` (30 days). Retention windows exposed as settings (`retention_concurrent_snapshots_days`, etc.) and gate uses a `retention_last_run_date` setting so the 60s cron tick is cheap when not due. Activity history is deliberately not pruned (user-visible data). **(6) Startup diagnostic** in `db.ts` logs `<file MB> on disk, <N> history rows, <M> concurrent_snapshot rows` after migrations so the operator can see scale trends without external tooling. Verified: lint clean on touched files (only pre-existing project-wide `any` warnings remain), `tsc` clean outside the known stale `.next/types` errors, all 7 vitest tests pass, migrations v2+v3 applied cleanly to the dev DB, EXPLAIN QUERY PLAN confirms index usage on all four hot queries. _Affected: `src/lib/migrations.ts`, `src/lib/db.ts`, `src/lib/db-types.ts`, `src/lib/history.ts`, `src/lib/user_stats.ts`, `src/lib/retention.ts` (new), `src/lib/cron.ts`._ Category: `refactor`
